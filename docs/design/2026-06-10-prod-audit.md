### [HIGH] sec-back/SEC-1 — Google OAuth links/creates accounts without checking email_verified
- **Where:** `backend/src/auth/oauth_google.rs:205`
- **Evidence:** The userinfo struct deserialises only `sub`, `email`, `name` (lines 82-87) — `email_verified` is never read. `upsert_oauth_user` then trusts the Google-asserted email unconditionally: step 2 links a Google identity to ANY pre-existing local user matching that email — `if let Some(row) = sqlx::query!(r#"select id from users where email = $1"#, info.email)... insert into oauth_identities (user_id, provider, subject)` — and step 3 creates a brand-new user with `email_verified_at = now()` straight from the unverified claim. No `email_verified == true` gate exists anywhere in the callback.
- **Impact:** Google's own 'Sign in with Google' guidance requires verifying the `email_verified` claim before trusting the email for account linking. An OAuth principal that can present an unverified email matching a victim's existing password account (e.g. a Workspace/Cloud-Identity account on an attacker-controlled domain, or any future federated IdP added here) is silently linked into that account, yielding takeover. It also mints local accounts pre-marked email-verified for addresses the requester may not own.
- **Fix:** Single file: backend/src/auth/oauth_google.rs. (1) Add `#[serde(default)] email_verified: bool` to the `Userinfo` struct (lines 82-87). (2) In callback(), immediately after parsing `info` (around line 144), reject when the claim is not verified: `if !info.email_verified { return Err(AppError::Validation("google email not verified".into())); }` — or, more conservatively, gate only the linking (step 2) and creation (step 3) branches in upsert_oauth_user so an existing linked identity (step 1) still works. Belt-and-suspenders: when creating a new account (step 3), only set `email_verified_at = now()` if `email_verified` is true. No migration or schema change needed; the data is already returned by Google's userinfo endpoint.

### [HIGH] corr-back/BE-1 — Replace endpoint accepts XISF but the pipeline cannot decode it — photo is bricked and the 7-day sweep then destroys the previous good original
- **Where:** `backend/src/photos/replace.rs:19`
- **Evidence:** ALLOWED_MIMES includes XISF: `const ALLOWED_MIMES: &[&str] = &["image/jpeg", "image/png", "image/tiff", "application/x-xisf"];` (replace.rs:15-20). The handler then runs `pipeline::finalize(..., PipelineOptions::Replace)` unconditionally (replace.rs:108-114). pipeline::finalize calls `thumbs::generate_blocking` → `image::load_from_memory(input)` (thumbs.rs:21-22), and the `image` crate has no XISF decoder, so the decode always fails and `queries::mark_failed` runs (replace.rs:119). Before the pipeline runs, the handler has already (1) enqueued the old original + old thumbnail keys into photo_pending_deletes (replace.rs:82), (2) swapped photos.storage_key to the new XISF (replace.rs:89-97), and (3) deleted the thumbnails DB rows (replace.rs:100-102). pending deletes are only drained on success (pipeline.rs:167-186), but `sweep_pending_deletes` deletes queued S3 keys unconditionally after 7 days regardless of photo state (jobs/purge_deletions.rs:60-78: `where queued_at < now() - interval '7 days'` then `storage.delete_objects(&stale)`). The XISF path that exists for primary uploads (upload_finalize.rs:69-81 branches to `mark_awaiting_calibration` + auto plate-solve) was never wired into replace.
- **Impact:** Any user who replaces a photo with a .xisf file gets a deterministically failed photo: status='failed', thumbnail rows gone, display master stale. Seven days later the hourly sweep permanently deletes the previous archival original and its thumbnails from S3 even though the replacement never became viable — irreversible data loss of the user's only full-resolution asset. The same loss applies to any failed replace (e.g. corrupt JPEG), XISF just makes it 100% reproducible.
- **Fix:** Three small changes. (1) backend/src/photos/replace.rs: either remove "application/x-xisf" from ALLOWED_MIMES, or mirror upload_finalize.rs:69-81 — after the storage-key swap, branch on mime == "application/x-xisf" to queries::mark_awaiting_calibration + platesolve_upload::auto_calibrate_xisf instead of pipeline::finalize. (2) backend/src/photos/replace.rs: add the magic-byte sniff (magic::sniff + magic::matches_mime, same as upload_finalize.rs:54-60) on the uploaded bytes BEFORE enqueue_pending_deletes/swap, so corrupt files are rejected up front and never reach the destructive prep. (3) backend/src/jobs/purge_deletions.rs: change the sweep SQL in sweep_pending_deletes to join photos and only delete rows where photos.status = 'ready' (e.g. "select d.storage_key from photo_pending_deletes d join photos p on p.id = d.photo_id where d.queued_at < now() - interval '7 days' and p.status = 'ready'", same predicate on the delete) — this preserves the documented retry semantics for healthy photos (pipeline.rs:159-161) while never destroying the prior original of a failed replace. Run cargo sqlx prepare and commit .sqlx/ after the SQL change.

### [HIGH] corr-back/BE-2 — Photo delete and account purge never delete the display master from S3 — deleted images remain publicly served by the CDN forever
- **Where:** `backend/src/photos/delete.rs:27`
- **Evidence:** delete.rs collects only the original and thumbnails: `let mut to_delete = vec![row.storage_key]; let thumb_keys: Vec<String> = sqlx::query_scalar!("select storage_key from thumbnails where photo_id = $1", id)...` (delete.rs:27-32). photos.display_key (`display/<id>.jpg` — the only key CloudFront serves) is never selected or deleted. The account purge has the identical gap: purge_one_user collects `select storage_key from photos` + thumbnails only (jobs/purge_deletions.rs:86-103), so every photo's display master AND the user's avatar object (`display/<avatar_id>.jpg`, users/avatar.rs:51-53) survive account deletion. Bonus leak: `delete from photos` CASCADE also removes photo_pending_deletes rows (comment at delete.rs:34-35), so any S3 keys queued by a pending replace are dropped from the cleanup queue without ever being deleted.
- **Impact:** Users who delete a photo — or their entire account (GDPR erasure flow with confirmation phrase and 7-day grace) — leave the full-resolution display JPEG publicly accessible at the CDN URL indefinitely. Anyone with the old URL keeps fetching the image from S3 origin forever. Also an unbounded storage-cost leak since display masters are the largest derived asset.
- **Fix:** 1) backend/src/photos/delete.rs: add display_key to the initial select (`select owner_id, storage_key, display_key from photos where id=$1`); `if let Some(dk) = row.display_key { to_delete.push(dk); }`. Also call crate::photos::queries::pending_deletes_for(&state.pool, id) before the row delete and extend to_delete with those keys (fixes the CASCADE-drop leak). 2) backend/src/jobs/purge_deletions.rs purge_one_user: add `select display_key from photos where owner_id=$1 and display_key is not null` to the key collection; add queued keys via `select pd.storage_key from photo_pending_deletes pd join photos p on p.id=pd.photo_id where p.owner_id=$1`; add the avatar object by reading users.avatar_id and pushing format!("display/{avatar_id}.jpg") when present. 3) Run `cargo sqlx prepare` from backend/ and commit .sqlx/. 4) Follow-up (separate concern): CloudFront invalidation for /img/<id>* on delete, since edge caches keep serving until TTL even after the S3 object is gone.

### [HIGH] corr-back/BE-3 — Plate-solve concurrency semaphore does not bound RSS: bodies are buffered and held before the permit is acquired
- **Where:** `backend/src/photos/platesolve_upload.rs:72`
- **Evidence:** The handler fully buffers the body before any concurrency control: `let parts = parse_multipart(multipart, id).await?;` (platesolve_upload.rs:72) under a `DefaultBodyLimit::max(MAX_XISF_BYTES)` of 128 MiB (http/mod.rs:428-435). The semaphore (`PLATESOLVE_MAX_CONCURRENT = 1`, http/mod.rs:23) is only acquired inside `run_solve` (`permits.acquire_owned().await`, platesolve_upload.rs:201) — i.e. after the handler has already spawned a task that owns the full `parts.xisf_bytes`. Queued tasks hold their 128 MiB bodies in memory while waiting for the single permit. The comments claim the opposite: "Acquire a semaphore permit so we cap concurrent solves (bounds RSS on the Koyeb tier)" (platesolve_upload.rs:12-13) and "this bounds RSS. Sized for the Koyeb Nano/Micro tier (≤512 MB)" (http/mod.rs:20-23). The auto-calibrate path has the same shape: it `storage.get`s the full XISF (plus a clone for render) before run_solve queues on the permit (platesolve_upload.rs:316-396).
- **Impact:** 3-4 concurrent 128 MiB platesolve POSTs (one user with several XISF uploads, or several users) exceed the 512 MB instance memory and OOM-kill the backend — a full outage, not just slow solves. The documented design invariant (semaphore bounds RSS) is violated by the implementation.
- **Fix:** Minimal fix, all in backend/src/photos/platesolve_upload.rs (plus a comment touch-up in backend/src/http/mod.rs):

1. Change run_solve to accept an already-held tokio::sync::OwnedSemaphorePermit parameter instead of permits: Arc<Semaphore> (delete the internal acquire_owned at lines 199-213; keep the explicit drop(permit) at line 276).

2. Side-channel handler: before `parse_multipart` (line 72), do `let permit = match Arc::clone(&state.platesolve_permits).try_acquire_owned() { Ok(p) => p, Err(_) => return Err(AppError::ServiceUnavailable / 503 with Retry-After) }` — try_acquire (not acquire) so a slow uploader can't pin the permit indefinitely and queued requests fail fast instead of buffering 128 MiB each. Move the permit into the spawned task and pass it to run_solve.

3. auto_calibrate_xisf: inside the spawned task, acquire the permit (`permits.acquire_owned().await` — blocking-queue semantics are fine here since nothing is buffered yet) BEFORE `state.storage.get(&storage_key)` at line 316, then pass it to run_solve. This makes queued auto-calibrates hold ~0 bytes instead of a full 50-200 MB file.

4. Update the now-stale comments: platesolve_upload.rs module doc lines 12-13 / run_solve doc lines 180-184, and http/mod.rs lines 18-23, to reflect that the permit is acquired before body buffering / S3 fetch.

Out of scope but worth a follow-up note: upload_finalize.rs:50 also buffers the full original with no aggregate bound (same class of issue for non-XISF finalizes).

### [HIGH] tests/TEST-1 — PAT (Bearer token) auth guards have zero integration test coverage — revoked-token rejection, admin-route rejection, and SessionOnly guard all untested
- **Where:** `backend/src/auth/middleware.rs:72`
- **Evidence:** The PAT resolution branch in resolve() — `select id, user_id from api_tokens where token_hash = $1 and revoked_at is null` (middleware.rs:72-75) — plus the two security guards built on it: AdminUser rejects PATs (`if parts.extensions.get::<TokenAuth>().is_some() { return Err(AppError::Forbidden); }`, line 174) and SessionOnly rejects PATs on account-control endpoints (line 192). `grep -rn 'Bearer' backend/tests/` returns ZERO hits — no integration test ever sends an Authorization: Bearer header. A silent regression — reordered extractor args, a refactor of resolve(), or dropping the `revoked_at is null` predicate — would let a revoked or stolen token change passwords/emails, delete accounts, mint new tokens, or hit /api/admin/* for an admin user. Nothing would catch it before production.

No production code changes required.

### [MEDIUM] sec-back/SEC-4 — Signup has no rate limiting or outbound-mail cap (account-creation / email-bombing abuse)
- **Where:** `backend/src/auth/signup.rs:63`
- **Evidence:** `handler` checks only the `signups_enabled` toggle, hashes the password, creates the user, then `email_verify::issue_token` + `mailer.send_plain(...)` on every request. There is no per-IP throttle, no global hourly cap, and no cooldown. By contrast `email_verify::resend` and `password_reset::request` both enforce `PER_HOUR_CAP`/`GLOBAL_HOUR_CAP` (e.g. email_verify.rs:30 `const GLOBAL_HOUR_CAP: i64 = 200`). The only backpressure on signup is the 8-permit Argon2 semaphore, which bounds concurrency, not volume.
- **Impact:** An attacker can mass-create accounts and send one 'verify your Astrophoto account' email to each of thousands of distinct victim addresses, harming SES sending reputation/cost and using the service as a mail-bombing relay — the exact abuse the reset/verify caps were added to prevent, left open on the highest-volume public mutation.
- **Fix:** In backend/src/auth/signup.rs, before calling email_verify::issue_token + mailer.send_plain, add the same site-wide hourly backstop the other mail paths use. Mirror email_verify::resend's global_cap_hit query (count rows in email_verification_tokens where created_at > now() - interval '1 hour' >= GLOBAL_HOUR_CAP); signup-issued tokens already feed that same bucket, so it needs only to be *checked*. On cap hit, skip token issuance + send (log a tracing::warn like the existing aggregate-cap warnings) while still returning the same verification_required response, or reject with AppError. Optionally expose GLOBAL_HOUR_CAP from email_verify (currently private) to reuse the constant. Per-IP throttling is intentionally avoided per the codebase's documented proxy-egress-collapse rationale (login_throttle.rs / password_reset.rs comments), so the global cap is the minimal consistent fix; a captcha gate is a larger optional follow-up.
