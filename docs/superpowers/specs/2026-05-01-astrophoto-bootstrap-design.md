# Astrophoto Bootstrap — Design

**Date**: 2026-05-01
**Status**: Approved (sections 1–3) — pending written-spec review
**Author**: Pascal (with Claude)

## Goal

Bootstrap a new project at `/Volumes/Pascal4Tb/Projects/astrophoto/` that
re-implements the astro-core features of the existing `astrodatasets`
Next.js app on a **Rust + SvelteKit** stack, and ships a `CLAUDE.md`
inspired by **Andrej Karpathy** (minimalism, "the code is the doc") and
**Boris Cherny** (the official Anthropic guidance: short, concrete, only
what cannot be inferred from reading the code).

Out of scope for the MVP: Stripe payments, BlockNote/Tiptap editor, OpenAI
integration, advanced 2FA. These can be added later behind the same
backend interface.

## Non-goals

- 1:1 feature parity with `astrodatasets`.
- Multi-region deployment, k8s, or auto-scaling.
- A "production-grade enterprise" framework. The CLAUDE.md is intentionally
  the opposite of the existing 760-line one.

## Backend architecture

- **Framework**: `axum` 0.7+ on `tokio` (multi-thread runtime).
- **DB access**: `sqlx` with the `query!` and `query_as!` macros, so SQL
  is checked at compile time. **No ORM.** **No Repository pattern.** SQL
  lives in `<feature>/queries.rs`.
- **Layers**: `http/handler` (thin, ~30 lines max each) → `<feature>/service`
  (orchestration, transactions) → `<feature>/queries` (SQL) → `db.rs` pool.
  No service trait abstraction with one impl.
- **Errors**: a single `AppError` enum (`thiserror`) implementing
  `IntoResponse` → JSON `{ error: "kebab-code", message: "...", request_id }`.
  Handlers return `Result<T, AppError>`. `anyhow` is forbidden in lib code.
- **Config**: `figment` reads env vars into a `Config` struct validated at
  boot. Missing required vars → panic immediately.
- **Tracing**: `tracing` + `tracing-subscriber`. JSON when not a TTY,
  pretty otherwise. Request IDs propagated.
- **Quality gates**: `cargo fmt --check`, `cargo clippy -- -D warnings`,
  `cargo test`. `clippy::unwrap_used` denied.
- **Type generation**: a small `bin/gen-types.rs` uses `ts-rs` to emit
  `frontend/src/lib/api/types.ts`. Run via `just types`.

## Frontend architecture

- **SvelteKit** with `adapter-node`. Svelte 5 runes only — no Svelte 4
  syntax (`export let`, `$:` reactive labels) is allowed.
- **`hooks.server.ts`** reads the session cookie, calls the Rust backend
  to resolve `event.locals.user`, attaches it for `load` functions.
- **Server `load`** in `+page.server.ts` fetches data via `lib/api/client.ts`.
  Client `+page.ts` is reserved for routes where data must be reusable on
  client-side navigation without refetching.
- **Form actions** for primary mutations (login, signup, upload). CSRF is
  automatic. No manual fetch for these.
- **Type safety**: API request/response types come from `lib/api/types.ts`
  (generated, never hand-edited).
- **No UI framework** at MVP. Vanilla CSS + variables. Add a kit when a
  third designer-touched page exists.

## Domain model (initial schema)

`backend/migrations/0001_init.sql` creates: `users`, `oauth_identities`,
`sessions`, `photos`, `thumbnails`. Key design points:

- **Sessions**: 32-byte random token stored as `bytea`, server-side. **No
  JWT.** Revocable via DELETE. Cookie is `__Host-session`, `Secure`,
  `HttpOnly`, `SameSite=Lax` (Lax is required for the OAuth redirect).
- **Photos**: EXIF dénormalized onto queryable columns (`taken_at`,
  `camera`, `iso`, `exposure_s`, etc.) **and** kept verbatim in a `jsonb`
  `exif_json` column.
- **Astro fields**: `ra_deg`, `dec_deg`, `target` (e.g. "M31"). Optional;
  populated when EXIF carries them or when the user adds them later.
  Plate-solving (offline pipeline producing `ra_center`, `dec_center`,
  `scale_arcsec_pix`) is **post-MVP**.
- **Indexes**: `(owner_id, created_at desc)` for galleries; `pg_trgm` GIN
  on `caption` for search; btree on `target`.
- **Status field** on `photos` (`ready | processing | failed`) drives the
  client-side UX during the spawn_blocking thumbnail pipeline.
- No soft-delete. `ON DELETE CASCADE` everywhere.

## Image pipeline

1. `POST /api/photos` accepts a multipart upload. Validate size (e.g.
   ≤ 50 MB), MIME (`image/jpeg`, `image/png`, `image/tiff`).
2. Stream the original to R2 under `originals/<photo_id>` while inserting
   a row with `status = 'processing'`.
3. `tokio::task::spawn_blocking`:
   - Parse EXIF via `kamadak-exif` → fill columns + `exif_json`.
   - Decode + resize via `image` crate → 400 px and 1200 px JPEGs → R2
     under `thumbs/<photo_id>/<size>`. Insert into `thumbnails`.
   - Update `photos` to `status = 'ready'` with `width`, `height`.
4. The frontend polls `GET /api/photos/<id>` until `status = 'ready'`
   (websocket later).
5. A periodic backend task re-scans rows stuck in `processing` for >10 min
   (the worker may have crashed) and marks them `failed`.

## Testing

- **Unit (backend)**: pure functions only — EXIF parsing, password
  hashing helpers, validation logic. No DB.
- **Integration (backend)**: `testcontainers` spins up Postgres per test
  run; the harness creates a fresh database per test. Covers auth flows,
  upload happy path, gallery queries.
- **Frontend**: `vitest` for `lib/`. **Playwright** for 2-3 happy paths
  (signup → login → upload → see thumbnail). No component-level tests at
  MVP.
- **CI**: GitHub Actions, single workflow, runs the same `just check` and
  `just test` users run locally. Postgres + MinIO services for integration.

## Deployment

- **Local dev**: `just dev` brings up `compose.yml` with `postgres:16` +
  `minio:latest` + backend + frontend, hot-reload for both.
- **Production**: single Hetzner VPS (CX22, ~€4/mo). `docker compose up -d`
  with the same services minus MinIO (R2 used instead via env vars).
  `caddy` reverse-proxies with auto-TLS.
- **Backups**: nightly `pg_dump` to R2 with 30-day retention.

## CLAUDE.md content (the core deliverable)

Target: ~180 lines, **zero emoji**, **no marketing fluff**, every line
passes the Cherny test "Would removing this cause Claude to make
mistakes?".

Sections, in order:

1. **Top** — one-paragraph orientation: what the project is, the stack,
   pointer to README and the spec dir.
2. **Philosophy** — 6 dense rules (Karpathy register):
   - the code is the documentation
   - simple beats clever
   - fail fast and loud
   - be slow, defensive, careful, paranoid (Karpathy verbatim)
   - YOU MUST NOT change code or comments you don't understand
   - IMPORTANT: keep diffs minimal, one concern per commit
3. **Repo map** — 6-line directory legend.
4. **Bash commands** — `just dev`, `just check`, `just test`, `just db`,
   `just types`, `just fmt`; backend-only (`cargo sqlx prepare`); frontend-
   only (`pnpm check`, `pnpm test:e2e`).
5. **Code style** — split into Rust and SvelteKit/Svelte 5 sub-sections.
   Rust: edition 2024, AppError + IntoResponse, no `.unwrap()` outside
   tests/main, no `anyhow` in lib code, `spawn_blocking` for blocking
   work, prefer `std::sync::Mutex`, `'static` requirement of
   `tokio::spawn`, sqlx macros for compile-time SQL, feature-folder
   layout, `tracing` only.
   Svelte: **runes only** (explicit ban on Svelte 4 syntax), reference
   to `https://svelte.dev/llms-small.txt`, prefer `$derived` over
   `$effect`, don't destructure into local consts, never export raw
   `$state`, server `load` for SSR routes, form actions for mutations,
   strict TS no `any`, generated types not hand-edited.
6. **Workflow** — `just check` must pass before claiming done; run
   single tests while iterating; `cargo sqlx prepare` after macro
   changes; `just types` after Rust API changes; migrations are
   append-only after merge.
7. **Repository etiquette** — branch naming, commit subject style, PR
   conventions, never `--no-verify`.
8. **Gotchas** — the 5 project-specific surprises that aren't inferable
   from code: `photos.status` async lag, EXIF naive datetimes, MinIO vs
   real S3 quirks, the `SameSite=Lax` cookie choice, Argon2 inside
   `spawn_blocking`.
9. **References** — design dir, svelte llms.txt, sqlx docs, agents.md.
